# Design Doc: 功能块 + 提示词生成（AI 驾驶舱 MVP）

## 概述
**需求来源**：产品转型——从代码可视化工具转为 AI 开发驾驶舱
**设计日期**：2026-04-28
**状态**：已确认

## 问题定义

PM 用 AI 开发项目时三个核心痛点：
1. 需求写不准 → AI 理解偏
2. 不知道功能在哪段代码 → AI 瞎找、漏链条
3. 项目迭代后功能链路不透明 → AI 另起炉灶

**根因**：AI 缺少项目级上下文约束。每次从零推断。

**解法**：在 PM 和 AI 之间插入"功能块"抽象层——PM 在功能粒度操作，系统翻译成精准的代码级约束提示词。

## 方案设计

### 推荐方案：功能块系统 + 一键提示词

**选择理由**：复用已有分组系统（Layer 4）作为自动生成基础，加上可编辑 + 命名 + 提示词输出。不需要造流程图编辑器，MVP 工作量可控。

---

#### 核心概念：功能块（Feature Block）

一个功能块 = 一组文件 + 一个业务描述 + 上下游关系

```
功能块 "数据获取" {
  文件: [fetcher.ts, source-scraper.ts, maccms.ts]
  描述: "从远端 URL 抓取 TVBox 配置源并做初步验证"
  上游: [config.ts (超时配置)]
  下游: [aggregator.ts, routes.ts]
}
```

#### 后端

**新增表 `feature_blocks`**：

```sql
CREATE TABLE feature_blocks (
  id TEXT PRIMARY KEY,
  project_id TEXT NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
  name TEXT NOT NULL,
  description TEXT,
  file_paths TEXT DEFAULT '[]',    -- JSON array
  is_auto INTEGER DEFAULT 1,       -- 1=自动生成, 0=手动创建/修改
  created_at TEXT DEFAULT (datetime('now'))
);
```

**新增/修改接口**：

| 接口 | 方法 | 功能 |
|------|------|------|
| `GET /api/projects/:id/blocks` | GET | 列出所有功能块 |
| `POST /api/projects/:id/blocks` | POST | 创建功能块 `{ name, description, filePaths }` |
| `PATCH /api/projects/:id/blocks/:blockId` | PATCH | 修改功能块 |
| `DELETE /api/projects/:id/blocks/:blockId` | DELETE | 删除功能块 |
| `GET /api/projects/:id/blocks/:blockId/prompt` | GET | 生成约束提示词 |

**自动生成逻辑**（扫描完成后）：

从现有 prefix clustering + 标注描述推断功能块：
1. 取 `file_groups` 表的分组结果作为初始候选
2. 对于未被分组的文件，按 role 聚类（所有 entry 文件一组、所有 config 文件一组）
3. 给每个块自动命名：从文件描述中取最频繁的关键词
4. 块的 description = 聚合子文件描述

**提示词生成逻辑** `GET /blocks/:blockId/prompt`：

```typescript
interface PromptResponse {
  prompt: string;       // 完整的约束提示词文本
  blockName: string;
  fileCount: number;
}
```

提示词模板：
```
## 功能模块：{name}
{description}

### 涉及文件（共 {n} 个）
{逐个列出：路径 — 行数 — 文件描述 — 角色}

### 上游依赖（本功能使用的外部模块）
{从 file imports 反推出的外部依赖列表}

### 下游影响（依赖本功能的外部模块）
{从 dependedBy 反推出的下游列表 + 影响文件数}

### 开发约束
- 修改范围限定在以上 {n} 个文件内
- 不要创建新文件实现已有功能
- 如需新增文件，放在 {directory} 目录下
- 改动后注意检查下游 {downstream_count} 个依赖文件
```

**功能块卡片组件 `FeatureBlockNode`**：

| 元素 | 说明 |
|------|------|
| 块名称 | 大字，如"数据获取" |
| 文件列表 | 紧凑列出包含的文件名 |
| 描述 | 一句话功能说明 |
| "生成提示词" 按钮 | 点击 → fetch prompt API → 复制到剪贴板 → 显示"已复制" |
| "编辑" 按钮 | 打开编辑模态框：改名/改描述/增删文件 |

**功能块编辑 Modal**：

```

│ ─────────────────────────────── │
```

**"生成提示词"点击后的交互**：

1. 按钮变为 loading 状态
2. fetch `GET /blocks/:id/prompt`
3. 弹出提示词预览 popover（可快速扫一眼内容）
4. 自动复制到剪贴板
5. 显示"已复制到剪贴板 ✓" 持续 2 秒

**视觉方向**：保持暗色仪表盘风格。功能块卡片比文件节点更大、更醒目——用实线边框 + 略亮背景（`bg-elevated`）区分于代码视图的节点。"生成提示词"按钮用 accent cyan 强调。

#### MCP 工具扩展

| 工具 | 功能 |
|------|------|
| `list_feature_blocks` | 列出项目的功能块 |
| `get_block_prompt` | 获取功能块的约束提示词 |
| `update_feature_block` | 修改功能块（AI 修订用） |
| `create_feature_block` | 创建新功能块 |

AI 开发者使用场景：扫描完毕后，通过 MCP 调用 `list_feature_blocks` 看看自动生成的是否合理，用 `update_feature_block` 修正不准的。

---

### 备选方案：功能流程图编辑器

**放弃理由**：流程图需要"顺序"概念（A→B→C），但代码里的调用关系往往不是线性的（A 同时调 B 和 C，C 又回调 A）。强加线性流程会失真。功能块的"上下游关系"已经从代码 import 自动计算出来了，不需要手动画箭头。

## 影响评估

- **涉及模块**：
  - 后端：新表 `feature_blocks` + 新路由 `routes/blocks.ts` + 修改 annotator 生成 blocks + prompt 生成逻辑
  - 前端：新组件 `FeatureBlockNode` + `FeatureBlockEditor` Modal + ProjectView 加视图切换 + 复制提示词交互
  - MCP：新增 4 个 tools
  - 不动：现有代码视图完全保留，功能视图是独立 tab

- **风险点**：
  - 自动生成的功能块可能不准（prefix clustering 的局限）→ 缓解：首次标注为"草稿"，PM 确认后变为"已确认"
  - 文件数多时功能块编辑可能卡 → 限制单块最多 20 个文件

- **技术债**：
  - 功能块间的"箭头"（上下游）目前是从 import 关系推算的，没有手动编辑能力 → 后续加
  - 提示词模板是硬编码的 → 后续可做成可配置模板

## 验收标准

1. ProjectView 有"代码视图"/"功能视图"切换
2. 功能视图展示自动生成的功能块卡片（含名称、描述、文件列表）
3. 点击"生成提示词"按钮 → 提示词复制到剪贴板
4. 提示词包含：文件列表、描述、上下游依赖、开发约束
5. 功能块可编辑（改名、改描述、增删文件）
6. MCP 工具可列出/修改功能块
7. 重新扫描后自动生成的功能块不覆盖已手动编辑的块
