# Design Doc: 分析引擎与模块地图 MVP

## 概述
**需求来源**：PRD 六大核心需求的前三项——加载代码识别模块、识别关联关系、整理数据给 AI 用
**设计日期**：2026-04-27
**状态**：已确认
**迭代范围**：第一迭代，用 tvbox-aggregator（95 文件，TS 单栈）验证全链路

## 问题定义

PM 有多个 AI 生成的项目，看不懂代码结构，无法做产品决策。需要一个工具：
1. 自动扫描项目 → 提取模块和依赖关系
2. 在浏览器里看模块地图（第一层：有什么；第二层：怎么连的）
3. AI 开发时通过 MCP 获取项目上下文，不再凭空编代码

第一迭代的成功标准：PM 打开浏览器能看到 tvbox-aggregator 的模块结构和依赖关系，AI 通过 MCP 能拿到同样的信息。

## 方案设计

### 推荐方案：TypeScript 全栈 + Tree-sitter + ReactFlow

**选择理由**：
- 首个目标项目是 TS，解析器生态天然匹配
- MCP SDK 是 TypeScript-first，免去跨语言胶水
- ReactFlow 节点就是 React 组件，定制 PM 友好的可视化成本最低
- 全栈同一语言，一个人就能维护

#### 后端

- **框架**：Fastify 5 + TypeScript
- **解析引擎**：web-tree-sitter (WASM，免 native 编译，Docker 跨架构无痛)
- **存储**：SQLite (better-sqlite3)——单文件部署，足够支撑多项目分析数据
- **运行时**：Node.js 22 LTS

**新增接口**：

| 接口 | 方法 | 功能 | 输入 | 输出 |
|------|------|------|------|------|
| `/api/projects` | GET | 获取所有已注册项目 | - | `Project[]` |
| `/api/projects` | POST | 注册新项目 | `{ name, gitUrl, token? }` | `Project` |
| `/api/projects/:id` | PATCH | 更新项目设置（含 token） | `{ token?, name? }` | `Project` |
| `/api/projects/:id` | GET | 获取项目详情 | - | `Project & { stats }` |
| `/api/projects/:id` | DELETE | 删除项目（清理 clone + DB 记录） | - | `{ success }` |
| `/api/projects/:id/scan` | POST | 触发重新扫描（git pull + 重新分析） | - | `{ jobId, status }` |
| `/api/projects/:id/scan/status` | GET | 查询扫描进度 | - | `{ status, filesTotal?, filesParsed? }` |
| `/api/projects/:id/modules` | GET | 获取模块列表 | `?depth=1` | `Module[]` |
| `/api/projects/:id/modules/:moduleId` | GET | 模块详情（含文件列表） | - | `Module & { files }` |
| `/api/projects/:id/dependencies` | GET | 获取依赖关系图 | - | `{ nodes, edges }` |

**数据模型**：

```sql
-- 项目
CREATE TABLE projects (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL,
  git_url TEXT NOT NULL,
  encrypted_token TEXT,          -- AES-256 加密存储，NULL 表示公开仓库
  local_path TEXT NOT NULL,      -- clone 后在容器内的路径
  default_branch TEXT DEFAULT 'main',
  last_scanned_at TEXT,
  scan_error TEXT,               -- 最近一次失败的错误信息
  created_at TEXT DEFAULT (datetime('now'))
);

-- 模块（一级目录 = 一个模块）
CREATE TABLE modules (
  id TEXT PRIMARY KEY,
  project_id TEXT REFERENCES projects(id),
  name TEXT NOT NULL,
  path TEXT NOT NULL,        -- 相对项目根的路径
  file_count INTEGER,
  line_count INTEGER,
  complexity_score REAL,     -- 0-100，基于文件数+行数+依赖数加权
  created_at TEXT DEFAULT (datetime('now'))
);

-- 文件
CREATE TABLE files (
  id TEXT PRIMARY KEY,
  module_id TEXT REFERENCES modules(id),
  project_id TEXT REFERENCES projects(id),
  path TEXT NOT NULL,
  language TEXT,
  line_count INTEGER,
  exports TEXT,              -- JSON: 导出的符号列表
  imports TEXT               -- JSON: 导入来源列表
);

-- 模块间依赖
CREATE TABLE dependencies (
  id TEXT PRIMARY KEY,
  project_id TEXT REFERENCES projects(id),
  source_module_id TEXT REFERENCES modules(id),
  target_module_id TEXT REFERENCES modules(id),
  weight INTEGER DEFAULT 1, -- 依赖强度（import 次数）
  UNIQUE(source_module_id, target_module_id)
);
```

**项目注册流程**：

```
用户操作（前端 UI）：
  1. 点击"添加项目"
  2. 填写：项目名称 + Git URL + Access Token（可选，公开仓库不需要）
  3. 点击"注册"

后端执行：
  1. POST /api/projects { name, gitUrl, token? }
  2. Token 加密存入 SQLite（AES-256，密钥来自环境变量 ENCRYPTION_KEY）
  3. Clone 时动态注入 token：
     git clone https://{project.token}@github.com/user/repo.git /data/projects/{name}/
  4. 自动检测默认分支（git remote show origin → HEAD branch）
  5. 自动检测主语言（按文件后缀统计）
  6. 自动触发首次分析
  7. 返回 project 对象 + scanJob 状态

扫描状态枚举：
  - cloning: 正在 clone/pull
  - parsing: 正在解析文件（返回 filesTotal + filesParsed）
  - analyzing: 正在计算依赖关系
  - done: 完成
  - error: 失败（返回 error message）

Token 管理：
  - 每个项目独立 token，支持不同 Git 平台（GitHub/GitLab/Gitea）
  - Token 过期时：前端项目设置页可随时更新
  - Clone/pull 失败时返回明确错误："认证失败，请更新 Token"

重新扫描：
  1. POST /api/projects/:id/scan
  2. git pull（用该项目自己的 token）
  3. 重新执行分析引擎
  4. 覆盖写入 SQLite
```

**Docker 配置**（docker-compose.yml）：
```yaml
services:
  backend:
    environment:
      - ENCRYPTION_KEY=${ENCRYPTION_KEY}  # 用于加密存储的 token
    volumes:
      - projects_data:/data/projects  # clone 的项目存这里
      - db_data:/data/db              # SQLite 文件

volumes:
  projects_data:
  db_data:
```

**核心逻辑——分析引擎流程**：

```
1. 从 /data/projects/{name}/ 读取已 clone 的代码
2. 读取 tsconfig.json → 解析 paths alias 映射表（如 @/* → src/*）
3. 扫描文件树 → 过滤目标语言文件（.ts/.tsx/.js/.jsx）
4. 按一级目录分组 → 生成模块
5. 对每个文件用 web-tree-sitter 解析：
   - 提取 import 语句 → 解析目标路径
   - 路径解析优先级：path alias → 相对路径 → 丢弃（node_modules 外部包）
   - 提取 export 语句 → 记录暴露的符号
   - 统计行数
6. 根据 import 的目标路径（仅项目内部），计算模块间依赖关系
7. 计算每个模块的复杂度评分
8. 事务写入 SQLite（全部成功 commit，失败 rollback 保留上次完整数据）
```

**Import 解析规则**：
- `import x from './foo'` → 相对路径，解析为实际文件
- `import x from '@/utils/bar'` → 读 tsconfig paths 映射为 `src/utils/bar`
- `import x from 'hono'` → 外部依赖，**丢弃**，不计入模块间关系
- `import x from '../services'` → 解析为 `services/index.ts`（不穿透 barrel，第一版已知限制）

**模块识别策略**（第一迭代）：
- 默认规则：项目根下的一级目录 = 一个模块
- 忽略：`node_modules/`、`.git/`、`dist/`、`build/`、隐藏目录
- 特殊处理：`src/` 下的一级子目录作为模块（如果项目根只有一个 `src/`）

#### 前端

- **框架**：Vite + React 19 + TypeScript
- **样式**：Tailwind CSS 4
- **图可视化**：@xyflow/react (ReactFlow v12) + elkjs (层次布局)
- **HTTP 客户端**：ky
- **状态管理**：Zustand（轻量，够用）

**页面/组件**：

| 页面/组件 | 功能 | 交互说明 |
|----------|------|---------|
| ProjectList | 项目列表 + 注册入口 | 选择项目进入地图视图 |
| ModuleMap | 模块地图主视图 | 节点=模块，边=依赖，支持缩放拖拽 |
| ModuleNode | 自定义 ReactFlow 节点 | 显示模块名、文件数、复杂度色块；点击展开文件列表 |
| DependencyEdge | 自定义 ReactFlow 边 | 粗细表示依赖强度，箭头表示方向 |
| ModuleDetail | 侧边栏详情面板 | 点击模块后显示：文件列表、导出符号、依赖列表 |
| ScanTrigger | 扫描触发按钮 | 点击后显示进度条，完成后刷新地图 |

**模块地图视觉设计**：
- 节点大小：按 line_count 映射（代码量越大节点越大）
- 节点颜色：按 complexity_score 映射（绿=简单，黄=中等，红=复杂）
- 边粗细：按 weight 映射（依赖越强线越粗）
- 布局：elkjs layered 算法（自上而下或自左而右，依赖方向清晰）
- 孤立模块（无依赖关系）单独区域展示，标记为"可能的死代码"

**与后端对接**：
- 页面加载 → `GET /api/projects` → 显示项目列表
- 进入项目 → `GET /api/projects/:id/dependencies` → 渲染地图
- 点击模块 → `GET /api/projects/:id/modules/:moduleId` → 侧边栏详情
- 触发扫描 → `POST /api/projects/:id/scan` → 轮询 status 直到完成

#### MCP Server（本地薄客户端）

- **SDK**：`@modelcontextprotocol/sdk` >=1.26.0
- **传输**：stdio（Claude Code 要求）
- **安装**：`npm install -g @code-atlas/mcp`（或 npx 运行）

**暴露的 Tools**：

| Tool 名称 | 功能 | 输入 | 输出 |
|-----------|------|------|------|
| `get_projects` | 列出所有项目 | - | 项目列表 |
| `get_project_overview` | 项目概览（模块+统计） | `{ projectId }` | 模块列表+整体统计 |
| `get_module_detail` | 模块详情 | `{ projectId, moduleId }` | 文件列表+导出符号 |
| `get_dependencies` | 依赖关系图 | `{ projectId }` | 全量依赖关系 |
| `get_impact_analysis` | 变更影响分析 | `{ projectId, moduleName }` | 依赖该模块的所有模块 |

**配置方式**：
```bash
claude mcp add code-atlas -- npx @code-atlas/mcp --api https://your-tunnel.domain.com
```

用户只需配一次，之后 AI 自动可用。NAS 通过 Cloudflare Tunnel 暴露，任何网络环境下 MCP 都能连通。

### 备选方案：Python (FastAPI) + 前端不变

**放弃理由**：
- MCP SDK Python 版成熟度低于 TypeScript 版
- 首个目标项目是 TS，用 Python 解析 TS 需要额外桥接
- 增加运维复杂度（Docker 里装两个运行时）
- 唯一优势是 Python 生态有更多分析工具（如 ast 模块），但这对 TS 项目没用

## 影响评估

- **涉及模块**：全新项目，无现有模块受影响
- **风险点**：
  1. **模块识别准确性**：一级目录 ≠ 业务模块，tvbox-aggregator 可能是技术分层目录。缓解措施：先跑一次看结果，不对再调策略
  2. **web-tree-sitter ABI 兼容性**：WASM grammar 文件必须与 web-tree-sitter 版本匹配。缓解措施：锁定版本，CI 验证
  3. **Barrel export 不穿透**：`import from '../services'` 解析到 index.ts 就停，不追踪 re-export 的具体来源。第一版已知限制，依赖图比实际粗
- **技术债**：
  1. 第一迭代只支持 TypeScript 解析，后续需要为每种语言写 ~50 行 tree-sitter query
  2. 模块识别策略硬编码为"一级目录"，后续需要支持自定义分组规则
  3. 无 Web 认证机制（通过 CF Tunnel 暴露，依赖 Tunnel 自身的 Access 策略保护）

## 验收标准

1. `docker compose up` 一条命令启动完整服务
2. 注册 tvbox-aggregator 项目后，触发扫描，30 秒内完成
3. 浏览器打开模块地图，能看到所有模块节点和依赖连线
4. 点击模块节点，侧边栏显示该模块的文件列表和导出符号
5. 节点大小/颜色正确反映代码量和复杂度
6. 在 Claude Code 中通过 MCP 调用 `get_project_overview`，返回与前端一致的模块信息
7. 孤立模块（无依赖）被视觉标记
