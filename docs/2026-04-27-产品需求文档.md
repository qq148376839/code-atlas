# code-atlas 产品需求文档

## 概述
**项目名称**：code-atlas（代码图谱）
**定位**：让不写代码的人也能看懂项目全貌的代码分析与可视化工具
**核心用户**：使用 AI 写代码的产品经理/项目负责人
**记录日期**：2026-04-27

## 核心痛点

用户有多个 AI 生成的项目，但自己不写代码。导致：

1. **看不懂项目全貌** → 无法作为产品经理规划下一步路径
2. **规划路径方向错误** → AI 跟着改了一堆屎山代码
3. **发现问题再返工** → 工作量指数级增加，引发更多 bug
4. **AI 缺少项目上下文** → 在不同设备/session 间切换时丢失项目理解

一句话总结：**先有地图，再谈路线；先看全貌，再碰细节。**

## 六大核心需求

### 1. 加载代码 → 识别模块
扫描项目代码，自动提取模块结构。识别前端页面/组件和后端服务/接口，而非只看后端。

### 2. 加载代码 → 识别关联关系
提取模块之间的调用依赖关系，包括：
- 前端页面 → 调用哪个 API
- API → 背后调用哪些服务/模块
- 模块之间的相互依赖

### 3. 整理数据 → AI 快速了解项目
将分析结果整理成结构化格式，AI 通过 CLI 加载后即可获得项目全貌上下文，无需重新阅读所有代码。

### 4. 加载后 → 快速识别潜在问题
标记代码中的风险点：死代码、循环依赖、过度耦合、安全隐患、代码质量问题等。

### 5. 关系链 → 约束 AI 不乱编
利用依赖关系图约束 AI 行为——修改 A 模块时，AI 必须知道会影响 B、C 模块，不能凭空编造不存在的接口或模块。

### 6. POST 方式 → 补充关键信息
支持对每个模块/节点按标准模板补充人工标注信息：
- 模块简介、关联性、关键要点
- 发现关键信息缺失时，可直接按模板补充
- 发现注释有误时，可按模板修订
- 持续沉淀，越用越准

## 模块地图（首要视图）

模块地图是产品经理的"作战地图"，必须回答三层问题：

### 第一层：有什么？
- 项目分成哪几个大模块？
- 每个模块多"重"？（代码量/复杂度的粗略感知）
- 哪些是核心模块，哪些是辅助模块？

### 第二层：怎么连的？
- 模块之间的调用关系（A 依赖 B？还是 B 依赖 A？）
- 哪个模块是"枢纽"——被最多模块依赖的
- 有没有孤立模块（没人连的，可能是废代码）

### 第三层：对应到业务
- 每个模块对应产品里的哪个功能/页面？
- 用户核心路径经过了哪些模块？
- 下个版本要改的需求，落在哪个模块上？

**关键要求**：前端和后端必须同时呈现。不是只看后端骨架——必须是"前端页面 → API → 后端逻辑"的完整链路。

## 后续视图（优先级递减）

| 视图 | 用途 | 什么时候看 |
|------|------|-----------|
| 依赖链路 | 改 A 会影响 BCD | 模块地图看完，准备动手改的时候 |
| 风险标注 | 哪些地方有坑、质量差、不能乱动 | 模块地图+依赖链路都清楚后 |
| 开发时间线 | 什么时候改了什么 | 出了问题需要追溯，或做复盘的时候 |

## 部署与架构

### 部署方式
- **Docker** 部署在用户的 NAS 上
- 一个统一平台，所有项目都在里面，切换着看
- 每个项目有自己的一套分析数据
- 支持任意设备、任意地点访问

### 系统组成

```
┌─ 分析引擎（后端）
│   ├→ 扫描项目代码 → 提取模块、函数、依赖关系
│   ├→ 生成结构化知识库（每个模块有标准模板）
│   └→ 支持多项目、多语言
│
├─ 前端可视化
│   ├→ 模块地图（三层）
│   ├→ 后续：依赖链路、风险标注、时间线
│   └→ POST 标注系统（在前端直接按模板补充/修正信息）
│
├─ CLI 接口
│   └→ AI 开发时调用，加载目标项目的知识库作为上下文
│
└─ Docker 部署
    └→ NAS 上跑，任意设备访问
```

### 使用场景
1. 打开浏览器 → 看到项目模块地图 → 理解全貌
2. 决定下一步做什么 → 基于地图做决策
3. 打开 Claude Code → AI 通过 CLI 拉取项目知识
4. AI 基于真实架构执行开发
5. 在前端查看结果 → 不对的地方 POST 修正
6. 知识库持续沉淀，越用越准

## 灵感来源：codeflow

参考项目 `/Users/rio/Documents/codeflow`——一个零配置的浏览器端代码可视化工具（6828 行单文件 index.html）。

**借鉴的思路**：
- AST 解析 + 正则回退的多层解析策略（Acorn/Babel/Tree-sitter）
- 函数调用关系检测（基于函数名匹配）
- 架构层级检测（UI/Service/Util/Config）
- 代码健康评分（A-F 等级）
- Blast Radius 分析（变更影响范围）
- 安全扫描（25+ 种问题检测）
- 设计模式识别

**不照搬的部分**：
- codeflow 是纯浏览器端，code-atlas 需要后端服务 + 持久化
- codeflow 重可视化酷炫，code-atlas 重信息密度和实用性
- codeflow 无标注系统，code-atlas 核心是人+AI 持续填充知识

## 技术栈偏好

用户不关心具体技术栈，只关心需求是否准确落地。选型交给 CTO 在 /spec 阶段决定。

约束条件：
- 必须支持 Docker 部署
- 前后端可以迭代
- 好运维、好扩展

## 第一个迭代的建议方向

> 看完模块地图后，产品经理的第一条指令不是"开始开发"，而是：
> "把这个项目拆成迭代计划，告诉我第一个迭代应该动哪个模块、为什么、风险在哪。"

建议从**分析引擎核心 + 最简模块地图**开始，先跑通"扫描项目 → 看到模块结构"的主路径。
